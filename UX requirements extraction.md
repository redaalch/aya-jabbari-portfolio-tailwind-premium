3. Target User Questions Each Section Must Answer

The hero must answer: Who is Aya? What does she do? Where is she based? Is she available? How do I contact her?

The about section must answer: What kind of engineer is she becoming? What kinds of problems does she care about? Why should I continue reading?

The skills section must answer: Does her stack match the role? Are her skills relevant to full-stack, backend, data, applied AI, and automation roles?

The experience section must answer: Has she worked in a real organization? What did she actually build at ONDA?

The hackathons section must answer: Has she applied AI to real document workflows? Has she shipped under time pressure?

The projects section must answer: What did she build? Why did she build it? Which parts did she own? Can a technical reviewer ask her about it in an interview?

The education section must answer: Where does she study? What is her current academic level?

The certification section must answer: Are her database and AI claims supported by third-party learning credentials?

The contact section must answer: What is the fastest way to reach her right now?

13. Hover, Focus, Loading, Empty, Success, and Error States
Hover states

Project cards should lift subtly, around 2–4px.

Cards can show a soft border or plum/pink accent.

Buttons should darken, fill, or slightly elevate.

Nav links can use a subtle underline animation.

Project images may scale slightly, but not aggressively.

Focus states

Every link, button, filter, and form input must have a visible focus state.

Focus rings should be high contrast, ideally plum on cream with offset.

Never remove outlines without replacing them.

Add a skip-to-content link.

Loading states

Core text should load first.

Images below the fold should lazy-load.

Use blur-up placeholders or fixed-size placeholders to prevent layout shift.

Avoid full-screen loaders.

Contact form submission can show “Sending…”

Empty states

No GitHub link: hide the button or show Code available on request.

No live demo: hide demo button.

No CV: hide Download CV until the file exists.

No screenshot: use a branded gradient placeholder or simple technical icon.

No certifications confirmed: show certificate names only, without verification links.

Success states

Copy email: Email copied.

Form submitted: Message sent. Aya will receive your message by email.

CV download: optional small toast, non-intrusive.

Error states

Form failed: show Sending failed — you can also reach Aya directly at ayajabbari81@gmail.com
.

Broken image: show fallback placeholder, not a broken-image icon.

Slow network: keep skeleton or placeholders; do not show scary errors too quickly.

14. Accessibility Requirements

The site should be accessible by default.

Requirements:

Use semantic HTML.

Use one <h1> only, preferably Aya’s name in the hero.

Use proper heading hierarchy: h2 for sections, h3 for project cards.

Use <header>, <nav>, <main>, and <footer> landmarks.

Maintain WCAG AA contrast.

Body text should be at least 16px.

Body line height should be at least 1.5.

Do not render text as images.

All images need descriptive alt text.

Decorative graphics should use empty alt text.

Icon-only links need accessible labels.

Do not rely on color alone for meaning.

All interactive elements must be keyboard accessible.

Project filters must be usable by keyboard and screen readers.

Modals must trap focus and return focus on close.

Respect prefers-reduced-motion.

No autoplay videos or carousels.

No animation should block access to content.

Final Design Principle

Every element on the page must pass two tests.

The recruiter test: does this help a non-technical recruiter decide to contact Aya in under 60 seconds?

The interview test: if a senior engineer asked Aya about this exact claim, could she defend it using her real CV and project experience?

If an element fails either test, redesign it or remove it.
